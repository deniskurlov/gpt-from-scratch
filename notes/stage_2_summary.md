# Stage 2: Token and learned positional embeddings

## Summary

This stage built the first lookup layers of the transformer: a `TokenEmbedding(V, d_model)` that maps the `(B, T)` int64 token IDs from stage 1's `get_batch` into a `(B, T, d_model)` float32 tensor, and a `LearnedPositionalEmbedding(T_max, d_model)` that maps a `(T,)` positions tensor into a `(T, d_model)` float32 tensor. Composed by elementwise sum, they produce the `(B, T, d_model)` representation that every downstream stage (attention, MLP, normalization, unembedding) operates on. The key conceptual contribution is the `d_model` dimension itself — the residual stream — into which every subsequent layer reads and writes. Without positional embeddings the attention layer in stage 3 would be permutation-equivariant (order-blind); the additive positional signal at the input is what makes sequence order matter at all.

## The math

**Embedding lookup ≡ one-hot matrix multiplication.** `nn.Embedding(N, d_model)(i)` is mathematically equivalent to `e_i^T · W` where `e_i ∈ R^N` is a one-hot vector and `W ∈ R^{N × d_model}` is the learnable embedding matrix. Both yield the i-th row of `W`. The indexed-lookup form is universal in practice for the same FLOP / memory reasons covered in stage 1's summary.

**Sum vs concatenation (the elementwise-sum justification).** For tokens and positions both encoded in `R^{d_model}`, concatenation `[tok; pos] ∈ R^{2d}` followed by a linear projection `W ∈ R^{2d × d}` yields `tok · W_t + pos · W_p` for the block decomposition `W = [W_t; W_p]`. This is the sum of two linearly-transformed copies — exactly the same operation class as elementwise sum, where the embedding matrices themselves play the role of `W_t` and `W_p` (both learned through backprop). So summing at the input is "concat + linear projection" already collapsed into one step, with zero loss of expressiveness. Concatenation would double the residual-stream width, quadrupling parameter counts across every downstream layer, for no functional gain.

**Permutation equivariance of self-attention (stage 3 motivation).** For a permutation `π` of input positions, `f_attn(π(x)) = π(f_attn(x))`. Output at position `i` depends only on the multiset of input vectors, not their order. Positional embeddings break this symmetry by attaching a position-specific signal *before* attention. Derivation sketch left implicit — formalized in stage 3.

## Design choices and why

- **`d_model = 128`.** A residual-stream width that's small enough to train quickly on M4 Pro / MPS but large enough to give the model meaningful representational capacity for character-level Shakespeare. Below ~64 the residual stream gets bandwidth-starved; nanoGPT-Shakespeare's default is 384, which is also fine; 128 is the sweet spot for a "trains in seconds-to-minutes per epoch" learning setup. Divisible by all reasonable head counts (2, 4, 8, 16) for stage 4.
- **`T_max = 256`.** Matches nanoGPT-Shakespeare conventions. Comfortable margin over likely training `T` values; enough to demonstrate the full forward pass without artificially constraining context. The hard cap is a property of learned positional embeddings specifically; RoPE (stage 13) will remove it.
- **Two separate `nn.Module`s, not one combined module.** Composition logic lives in the caller (eventual `GPT` class), which is fine — the alternative of bundling both into one `Embeddings` module would (a) couple swapping positional schemes to internal restructuring, (b) prevent unit-testing each module in isolation. The split also pays off in stage 13: the RoPE migration deletes the `LearnedPositionalEmbedding` entirely and moves positional logic inside attention, which is cleaner with the module separated than embedded.
- **`pos_emb.forward(positions: Int64[Tensor, "T"])`, not `forward(T: int)`.** Caller passes an explicit positions tensor. More flexible: stage 12's KV cache will call `pos_emb(torch.tensor([T_so_far]))` for one-new-token-at-a-time inference, which the int-arg form wouldn't support naturally. Also moves device responsibility to the call site where the input device is already known.
- **Wrap `nn.Embedding` rather than re-implement from `nn.Parameter` + indexed lookup.** Same outcome, fewer lines, zero pedagogical loss — the math was already covered in stage 1. CLAUDE.md's "implement primitives yourself" rule applies to layers with hidden complexity (attention, LayerNorm, cross-entropy); `nn.Embedding` is a learnable tensor plus a lookup, with nothing inside worth touching by hand.
- **Elementwise sum for composition.** Justified above; concatenation buys no expressiveness for ~4× the parameter cost downstream.
- **Tests parametrize over `T ∈ {1, 8, 32, 128, 256}`** instead of testing one representative value or iterating over the full range. Boundary cases (1, T_max) plus representative interior values; failure granularity preserved via pytest's per-T test naming.

## Errors and corrections

Synthesized from `notes/stage_2_embedding_concepts.md`:

- **Called self-attention "permutation-symmetric".** Strict term is **permutation-equivariant**: `f(π(x)) = π(f(x))`. Permutation-invariant would be `f(π(x)) = f(x)`, which is strictly weaker and *not* what attention has. The equation written was right; the name attached to it was off.
- **Vague on sum vs concatenation.** First attempt caught "wider model is slower to train" but missed (1) parameter count downstream roughly quadruples (not just doubles at the input — every subsequent linear layer scales with the residual-stream width), and (2) the mathematical-equivalence argument: concatenation + linear ≡ elementwise sum + (the embedding matrices acting as projections). Sum is strictly cheaper for the same expressiveness.
- **Suggested `d_model = 12-16` from Johnson-Lindenstrauss.** JL gives the *information-theoretic minimum* for distinguishing V categorical tokens, but `d_model` is the **residual-stream dimension** — the bandwidth of every block's read/write to the shared representation, not just an embedding dimension. The model needs `d_model` capacity for syntactic role, position information, attention-output accumulation, and everything else it computes per layer. Choking it at 16 would crush the model's compute, not just its token storage. Anthropic-mechanistic-interpretability framing (Olah, Elhage) was the missing piece of intuition.
- **`__inti__` typo, twice in a row.** Once in the commented-out version, once after I explicitly flagged it. Python doesn't complain (any method name is valid; `__inti__` is just a regular attribute). The default `__init__` from `nn.Module` runs instead, `self.pos_emb` never gets created, and the bug surfaces at first call as `AttributeError`. Magic method names are strings Python looks up at specific moments; typos turn dunder methods into dead code.
- **`super.__init__()` instead of `super().__init__()`.** `super` (no parens) is the class; `super()` is the bound instance. Same kind of distinction as `dict` vs `dict()`. The parenless form raises at runtime.
- **`TokenEmbedding(V=tok.vocab, d_model=128)` in `__main__`.** Passed the vocab *list* (`tok.vocab`) instead of the vocab *size* (`tok.vocab_size`). `nn.Embedding` expects an integer for its first argument; PyTorch's error message buried the actual cause under "received an invalid combination of arguments".
- **Treated the embedding output's runtime size as the parameter count.** Initial guess: `token_emb` has `B · T · d_model` parameters. Wrong — parameters are the entries of the learnable table itself, shape `(V, d_model)`, independent of batch. Parameters vs activations is one of the most important distinctions to internalize early: parameters live on the model (set by architecture, persistent), activations live per-forward-pass (set by input shape, transient).
- **Slight imprecision in line-by-line walkthrough.** Two corrections: `super().__init__()` is about *parameter / submodule tracking*, not method inheritance (methods are inherited automatically by subclassing). `nn.Embedding(V, d_model)` is a *module wrapping* a learnable tensor (`.weight`), not the tensor itself.
- **Test bug: referenced fixture `text` without declaring it as a parameter.** Pytest only injects fixtures whose names appear in the test function's parameter list; references outside that resolve to the fixture function definition itself, which is non-iterable. `'FixtureFunctionDefinition' object is not iterable` — the diagnostic message gave away the cause.

## What this enables

- **Stage 3 (scaled dot-product attention, single head, with causal mask)** consumes the `(B, T, d_model)` float32 tensor produced by stage 2 — three `nn.Linear` projections give Q, K, V each of shape `(B, T, d_model)`. The causal mask is what retroactively justifies stage 1's shift-by-1 supervision (`B·T` parallel training examples become *honest* once future positions are masked out of attention scores).
- **Stage 4 (multi-head attention)** reshapes the `(B, T, d_model)` representation into `(B, n_heads, T, head_dim)` where `head_dim = d_model / n_heads`. The choice of `d_model = 128` already supports `n_heads ∈ {1, 2, 4, 8, 16}` cleanly.
- **Stage 7 (transformer block)** stacks attention + MLP + LayerNorm around the residual stream, both reading from and writing back to the `(B, T, d_model)` tensor that originated here. Every layer sees this representation; every layer is bounded in its expressiveness by `d_model`.
- **Stage 13 (RoPE migration)** deletes `LearnedPositionalEmbedding` from the model and adds rotation logic inside attention. The clean modular separation built here makes that swap structural rather than invasive.
