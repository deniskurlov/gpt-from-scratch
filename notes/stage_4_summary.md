# Stage 4: Multi-head attention

## Summary

This stage built `MultiHeadAttention` тАФ single-head causal attention (stage 3) generalized to `n_heads` parallel attention computations operating on the same `d_model`-dim residual stream. Mechanically, multi-head is one extra dimension of size `n_heads` inserted into the attention computation (via reshape/view), one new scaling factor `тИЪhead_dim` (where `head_dim = d_model / n_heads`), and one output projection `W_O` that is now structurally load-bearing rather than vestigial. The substantive content of the stage isn't algorithmic тАФ it's about *why* multiple heads matter: each head learns its own attention pattern (QK circuit) and its own write direction (OV circuit), so `n_heads` heads give the model `n_heads` independently-specializable routing primitives per layer. Without multi-head, the model could only commit to one attention pattern per layer per position, fundamentally limiting what kind of compositional information routing the network can learn. With multi-head, the architecture provides the structural inductive bias for specialization; SGD then finds specialized solutions (induction heads, name-mover heads, duplicate-token heads тАФ the named circuits in mechanistic-interpretability literature) because they're lower-loss than smeared single-pattern alternatives.

## The math

**Per-head scaling factor.** The same variance argument as stage 3, but applied per-head: for each head, `Q_h, K_h` are `head_dim`-dimensional, so `Var(Q_h ┬╖ K_h) = head_dim`. The pre-softmax scaling is `1/тИЪhead_dim`, not `1/тИЪd_model`. For single-head with `d_k = d_model` they coincide; for multi-head they differ by `тИЪn_heads`. The same softmax-saturation argument applies тАФ unit-variance pre-softmax scores keep gradients flowing through softmax at initialization.

**QK circuit, OV circuit (per head).** For each head `h`:
- **QK circuit**: `W_Q^h^T ┬╖ W_K^h` is a `(d_model ├Ч d_model)` bilinear form, rank тЙд `head_dim`. Determines the attention pattern (where head `h` attends from each position).
- **OV circuit**: `W_V^h ┬╖ W_O^h` is a `(d_model ├Ч d_model)` linear map, rank тЙд `head_dim`. Determines what information head `h` writes to the residual stream when attending.

Total attention output is the sum over heads of `(attention_pattern_h) ┬╖ (OV_circuit_h)` applied to the input. The decomposition is what makes mechanistic-interpretability analysis tractable.

**Why `W_O` is structural in multi-head.** Without `W_O`, each head's output would live in its own fixed `head_dim`-wide subspace of `d_model` (determined by the concat order). `W_O` is what lets each head's output get written to *any* direction in the residual stream тАФ it carries the per-head "write direction" parametrization (each `(d_model, head_dim)` block of `W_O` is paired with one `W_V^h` to form one head's OV circuit). In single-head this collapsed to a vestigial `(d_model, d_model)` projection foldable into `W_V`; in multi-head, `W_O` is essential and not foldable into per-head `W_V`s without changing the total number of parameters.

## Design choices and why

- **Add `MultiHeadAttention` alongside `Attention`** rather than replacing the single-head class. Both classes coexist for pedagogical clarity: the stage 3 implementation stays available as the simpler reference, while stage 4 adds the multi-head version. Production codebases sometimes merge them (single-head as `n_heads=1` of the multi-head class), but for a learning project preserving both makes the diff between them inspectable.
- **`(T_max, n_heads, d_model)` constructor**, with `head_dim` derived. Three architectural knobs; `head_dim` is a *consequence*, not a *parameter*. Avoids the d_k тЙа d_v generalization that's never used in practice. The `assert d_model % n_heads == 0` catches misconfiguration at init rather than letting it cascade to a confusing shape error 100 lines later.
- **Fused QKV** as `nn.Linear(d_model, 3*d_model)`. Same total parameter count as three separate `nn.Linear(d_model, d_model)` projections, better memory locality, one kernel call instead of three.
- **Split, then reshape per-component**: `qkv.split([d_model, d_model, d_model], dim=-1)` produces three `(B, T, d_model)` tensors, then each is reshaped via `.view(B, T, n_heads, head_dim).transpose(-2, -3)`. Alternative тАФ `view(B, T, 3, n_heads, head_dim).permute(2, 0, 3, 1, 4).unbind(0)` тАФ is one fewer step but uglier and less debuggable. The split-then-reshape pattern keeps each tensor's shape transformation legible.
- **`view` + `transpose` for the forward reshape; `transpose` + `reshape` for the reverse**. `view` is fine going in because `nn.Linear` output is contiguous. The post-`attn @ v` tensor is contiguous (matmul output), but `transpose(-2, -3)` makes it non-contiguous, so the reverse merge must use `reshape` (which falls back to a copy as needed) rather than `view` (which would raise). One O(N) copy at the merge is unavoidable in this implementation; FlashAttention-style fused kernels eliminate it.
- **`transpose(-2, -3)` and `transpose(-1, -2)`** with negative indices throughout. Forward-compatible across rank-3 (stage 3 single-head) and rank-4 (stage 4 multi-head) representations; same expression works for both.
- **`тИЪhead_dim` scaling**, not `тИЪd_model`. Written in code as `sqrt(self.head_dim)`. Variance argument applies per-head; using `d_model` would over-scale and produce softmax inputs of unit-variance / `тИЪn_heads` тЙИ smaller-than-needed.
- **Same causal mask** (T_max, T_max) bool, broadcasts against (B, n_heads, T, T) scores automatically. No mask modification needed тАФ broadcasting is the standard PyTorch idiom and applies the same mask to every head and every sequence in the batch. Causal structure is a property of positions, not heads.
- **All `Attention` tests mirrored to `MultiHeadAttention`** with parallel structure. 5 + 5 = 10 tests. The causality test in particular is the structural check тАФ it would fail immediately if the mask were misapplied per-head (e.g., wrong shape, wrong axis).

## Errors and corrections

- **"Single head must commit" intuition gap.** Initial answer to "why multi-head" was the standard "different heads can specialize" framing without operationally feeling why 4-8 heads is enough or why a single head couldn't learn all patterns. Sharpened: a single head computes ONE attention distribution per position тАФ mathematically forced to weighted-average all gathered values together. Multi-head writes to separate subspaces, then `W_O` mixes тАФ structurally separable. The "feel" comes from inspecting trained attention patterns in TransformerLens; theoretical scaffolding only gets you so far.
- **`split` vs `reshape` confusion.** First reshape attempt was `(B, T, d_model).split([n_heads, head_dim], dim=-1)`. `split` returns *separate tensors*; the right tool for "factor a dim into two factor-dims" is `view`/`reshape`. Distinct operations: `split` cuts a dim (produces N tensors), `view`/`reshape` refactors a dim (produces 1 tensor with different shape). Same naming-adjacency confusion class as `chunk` vs `split` from stage 3.
- **`assert` placement before `super().__init__()`** in `__init__`. Functionally fine in this case (the assert doesn't reference `nn.Module` state), but convention is `super().__init__()` first. Fixed on review.
- **`x.view(n_heads, head_dim)` missing leading dims** тАФ would reshape the entire tensor to 2-D. Needed `view(B, T, n_heads, head_dim)`. Caught during the snippet review.
- **Confusing `view` with in-place.** `.view()` returns a new tensor (with shared storage but different metadata) тАФ not in-place. PyTorch convention: `_` suffix means in-place (`add_`, `mul_`, `transpose_`); no suffix means functional (returns new tensor). `view` has no suffix тЖТ functional.
- **Forgot to store `self.d_model`** in `__init__` while referring to it in `forward` тАФ would have raised `AttributeError` at first call. Fixed by adding `self.d_model = d_model` to `__init__`.
- **`x.shape[:1]` instead of `x.shape[:2]`** for `B, T = ...` unpacking. `[:1]` gives a 1-tuple; the unpack to two names would have ValueError'd. Simple slice-bound typo.
- **`super().__init__()` "initializes parent's parameters" walkthrough imprecision** тАФ *recurring* (same imprecision in stage 3). The correct framing: `super().__init__()` initializes parent *instance state* (the `_parameters`, `_modules`, `_buffers` OrderedDicts), not "parent parameters". `nn.Module` has no parameters of its own; what it has is the machinery to track parameters added by subclasses.
- **Toy prediction: `self.mask[:T, :T]` shape predicted as `(1, 1, T, T)`** тАФ actual is `(T, T)`. Confused broadcast-imputed shape with actual shape. Tensor's `.shape` is `(T, T)`; broadcasting treats missing leading dims as size 1 *for the operation*, but the tensor itself is `(T, T)`.
- **Toy prediction: `scores.masked_fill(...)` output as `(B, n_heads, T, head_dim)`** тАФ actual is `(B, n_heads, T, T)`. Same error class as stage 3's "softmax drops a dim" mistake: shape-preserving operations preserve shape. `masked_fill` only replaces values, doesn't change the shape.
- **Test bug: `test_mha_total_parameter_count` initially tested only `qkv_proj.parameters()`** rather than `mha.parameters()`. The name said "total" but the assertion was partial. Fixed to use `mha.parameters()` and `4 * d_model * (d_model + 1)` formula (qkv_proj + out_proj together).

## What this enables

- **Stage 5 (LayerNorm)** wraps the residual stream around attention and MLP. Doesn't change attention internally тАФ but the `(B, T, d_model)` interface of `MultiHeadAttention` is exactly what LayerNorm operates on.
- **Stage 6 (MLP / pointwise FFN with GELU)** is the per-token feature-processing half of every transformer block. Stage 4's multi-head attention is the position-mixing half; together they form the two-component structure of every transformer block.
- **Stage 7 (transformer block)** stacks `MultiHeadAttention + MLP` with `LayerNorm` (pre-norm convention) and residual connections. Stage 4 is the position-mixing core; stage 7 is the assembly.
- **Stage 8 (full GPT)** stacks `n_layers` transformer blocks, plus token+positional embeddings (stage 2) and a final LayerNorm + unembedding head. Multi-head attention is reused identically per layer.
- **Stage 12 (KV cache)** is built directly on top of multi-head attention. K and V are cached across generation steps so each new token's attention only needs to compute one new Q and one new (K, V) pair, not recompute all of K, V for the whole context. The per-head decomposition is what makes the KV cache structure clean тАФ each head caches its own K, V.
- **Stage 13 (RoPE)** replaces the additive positional embedding with a position-dependent rotation applied to Q and K *inside* multi-head attention. Per-head application тАФ each head's Q and K get rotated independently. Stage 4's per-head structure is what makes RoPE's per-head action natural.
- **Stage 14 (GQA)** generalizes the per-head structure: fewer K, V heads than Q heads (key-value heads shared across query-head groups). A direct refactor of stage 4's `MultiHeadAttention` where the `n_heads` split is asymmetric between Q vs K/V.
